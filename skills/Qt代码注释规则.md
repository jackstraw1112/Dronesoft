# Qt代码注释规则（Dronesoft）

> 目标：让注释“解释原因与约束”，而不是重复代码字面含义。  
> 适用范围：`source/` 与 `source/01-TaskPlan/` 下 `*.h`、`*.cpp`、`*.ui` 相关代码。  
> 核心原则：注释应提升维护效率、降低误解成本、减少回归风险。

## 1. 注释总原则

1. 注释描述 **Why（为什么）**、**约束条件**、**副作用**、**边界行为**。
2. 不写“翻译型注释”（代码已明确表达，不再逐字重复）。
3. 注释必须与代码同步修改，禁止“过期注释”。
4. 注释简洁、准确、可执行，避免模糊词：`可能`、`应该`、`大概`。
5. 统一中文注释（项目现状为中文为主）。

## 2. 什么时候必须写注释

以下场景必须补注释：

1. **复杂业务判断**：多分支条件、策略选择、阈值判断。
2. **Qt 生命周期/所有权关键点**：父子对象销毁关系、手动 `delete` 原因。
3. **信号槽设计意图**：为何在此处连接、为何用 lambda、为何阻断重入。
4. **UI/数据同步逻辑**：表格行号与数据索引映射关系、状态刷新时机。
5. **临时兼容逻辑**：兼容旧字段、迁移过渡方案（需标注后续清理条件）。
6. **性能敏感处理**：批量更新、避免重复刷新、延迟渲染等。

## 3. 什么时候不要写注释

以下情况不要写注释：

1. 代码本身已经足够清晰。  
   例如：`count++` 不需要注释“计数加一”。
2. 注释只重复函数名。  
   例如：`// 初始化对象` + `initObject()`（若函数已很短可省略）。
3. 为了“凑注释量”而写无信息增量文本。

## 4. 注释位置与格式

1. **函数级注释**：放在函数定义前一行，说明职责、关键输入输出、副作用。
2. **代码块注释**：放在复杂分支/关键语句块前，不要行尾堆叠长注释。
3. **单行注释风格**：统一使用 `//`。
4. **多行说明**：使用连续 `//`，不使用大段 `/* ... */`。
5. 注释与代码保持相同缩进层级。
6. 注释密集区域要**适当留空行分段**（建议在不同业务步骤之间空 1 行），让代码形成可视段落，提升可读性。
7. 同一段落内保持“注释 -> 代码块”紧邻，不要在注释与其直接作用语句之间插入无关代码。

## 5. 推荐注释模板

### 5.1 函数注释模板

```cpp
// 设置选中的任务：更新卡片高亮并向外发射选中信号。
// 边界：当 taskId 不存在时直接返回，不改变当前状态。
void setSelectedTask(const QString &taskId);
```

### 5.2 关键分支注释模板

```cpp
// 删除后立即清空 selectedTaskId，避免外部槽函数在 deleteTaskClicked
// 回调中访问已删除卡片导致悬空引用。
m_selectedTaskId.clear();
```

### 5.3 数据映射注释模板

```cpp
// row 与 m_pointTargets 索引一一对应，插入/删除必须同步执行。
ui->pointTargetTbw->insertRow(row);
```

## 6. Qt专项注释建议

### 6.1 信号槽

1. 在 `connect(...)` 附近说明连接目的（尤其跨组件连接）。
2. 对“防重入、防循环触发”逻辑加注释说明触发链路。

### 6.2 UI对象与布局

1. 解释控件尺寸策略的原因（如 `Stretch`、`Fixed` 的取舍）。
2. 说明动态启用/禁用按钮的状态来源（例如“基于当前行是否有效”）。

### 6.3 内存与对象所有权

1. 明确 QObject 父子关系托管还是手动释放。
2. 如果出现 `new` + `delete`，需要注释“为何不能交由父对象自动销毁”。

## 7. 注释质量标准（提交前自检）

每条注释至少满足以下之一：

1. 解释业务意图（Why）；
2. 解释边界行为（Boundary）；
3. 解释风险/副作用（Risk/Side Effect）；
4. 解释实现约束（Constraint）。

若都不满足，建议删除该注释。

## 8. 反例与正例

### 反例

```cpp
// 把按钮设为不可用
ui->btnDeleteTask->setEnabled(false);
```

### 正例

```cpp
// 删除按钮仅在存在有效选中行时启用，防止误触发空删除流程。
ui->btnDeleteTask->setEnabled(currentRow >= 0);
```

## 9. 与现有规范协同

1. 命名与注释配合：变量名先表达“是什么”，注释补充“为什么这样做”。
2. 参考文档：
   - `skills/Qt代码命名规则.md`
   - `skills/Qt控件缩写规则.md`
3. 注释不替代良好命名；先改名，再考虑补注释。
4. `objectName` 示例统一使用“控件缩写在前、业务语义在后”（如 `btnSaveTask`、`leTaskName`、`tbwPointTarget`）。

## 10. 渐进落地建议

1. 新代码严格执行本规则。
2. 修改旧代码时按“触点清理”补注释，不做一次性全量重写。
3. Code Review 时重点看：
   - 复杂逻辑是否解释了原因；
   - 注释是否已过期；
   - 是否存在“翻译型注释”。

## 11. 注释粒度分档（详细 / 适度 / 简要）

> 本项目注释分三档，评审和开发时需先选档再写注释。  
> 默认档位：**适度**。涉及关键业务流程、跨模块交互、高风险逻辑时使用**详细**。

### 11.1 详细档（High Detail）

适用场景：
1. 核心业务流程（保存、提交、删除、状态推进）。
2. 涉及多步校验、状态机、跨信号链路。
3. 历史兼容逻辑、迁移过渡逻辑、容易误改逻辑。

要求：
1. 按“阶段/步骤”组织注释（如“第一阶段、第二阶段”）。
2. 明确写出：**为什么这样做**、**不这样做的风险**、**顺序约束**。
3. 关键分支与关键语句都要有解释。

### 11.2 适度档（Medium Detail）

适用场景：
1. 普通业务函数（有少量分支和边界）。
2. UI 与数据同步函数、常规增删改逻辑。

要求：
1. 函数前写 1~2 行职责说明。
2. 复杂分支前补 1 行注释，说明意图或边界。
3. 不要求每一步都注释，避免噪声。

### 11.3 简要档（Low Detail）

适用场景：
1. 逻辑非常直白的 getter/setter、轻量转发函数。
2. 命名已经充分表达语义的短函数。

要求：
1. 可不写注释，或仅保留 1 行背景说明。
2. 严禁“翻译型注释”。

### 11.4 选档决策（快速判断）

满足任一项即升级到**详细档**：
1. 会修改持久化数据或触发保存/删除。
2. 会发出多个信号且存在顺序依赖。
3. 有历史兼容分支或潜在回归风险高。

都不满足但有边界条件时使用**适度档**；  
纯直线、低风险逻辑使用**简要档**。

### 11.5 三档对照示例（同一语句）

```cpp
// 简要档：更新状态。
```

```cpp
// 适度档：根据当前选中行更新删除按钮可用性，避免空删除。
```

```cpp
// 详细档：删除按钮仅在存在有效选中行时启用。
// 这样可避免误触发删除流程导致“空对象删除”提示干扰，同时保持
// 与双击编辑/右键菜单的可操作状态一致，降低用户误操作概率。
```
